# 메타데이터 태그 규격 (ID3 / Vorbis Comment)

## 규격별 기술 사양

### ID3 (v1 / v2.3 / v2.4)
* 주로 MP3 파일 등 고정식 오디오 포맷의 앞머리나 끝자락 헤더 영역에 앨범명, 가수 이름, 발매 연도, 가사 정보, 고해상도 이미지 바이트를 텍스트 규격으로 밀어 넣어둔 메타데이터 표준입니다. 
* ID3v2.3은 호환성이 가장 무난하고, ID3v2.4는 최신 규격으로 UTF-8 유니코드를 기본 활용하여 전 세계 한글/일본어 폰트 깨짐 문제를 방지합니다.

### Vorbis Comment
* FLAC 및 Ogg Vorbis 파일 등에서 사용되는 메타데이터 표준으로, 고정식 ID3 프레임 구조와 달리 `FIELD=Value` 형태의 유연한 사용자 텍스트 페어로 태그를 기록합니다. 구조가 극도로 심플하여 태그 스캔 속도가 빠르고 앨범 자켓 이미지도 고속으로 헤더에서 추출해 플레이어 화면에 띄워줍니다.

## ⚠️ 메타데이터 파싱 불일치 및 오류 원인

UAPP나 고음질 재생기에서 폴더를 불러올 때 곡 정보 인식이 들쭉날쭉한 주요 원인은 다음과 같습니다.

* **내장 태그(Embedded) 유무**: 파일 내부 헤더(Vorbis Comment, ID3 등)에 앨범명, 아티스트, 트랙 정보가 제대로 기입되어 있지 않으면 플레이어는 파일 이름(예: `01 - 곡제목.flac`)이나 상위 폴더 이름에서 정보를 추정하여 파싱을 시도합니다. 이 과정에서 구분자(하이픈, 띄어쓰기) 오차로 인해 트랙이 분할되거나 꼬이게 됩니다.
* **텍스트 인코딩 불일치**: 과거 한글 음원 리핑 시 유니코드(UTF-8)가 아닌 구형 코드페이지(EUC-KR 등)로 태그가 작성된 경우, 플레이어가 이를 디코딩하지 못해 글자가 깨지거나 아예 메타데이터 파싱 자체가 실패합니다.

## 📂 폴더 내 기타 파일이 파싱에 미치는 영향

음원 폴더 내부에 포함된 부가 파일들은 음악 재생 프로그램의 동작과 매핑에 중요한 작용을 합니다.

* **이미지 파일 (`cover.jpg`, `folder.jpg`, `AlbumArt.jpg`)**: 음원 자체에 자켓 이미지가 내장되어 있지 않더라도, 폴더 내에 이 같은 표준 파일명으로 이미지 파일이 존재하면 플레이어가 이를 앨범 아트로 인식하여 대표 커버로 표시합니다.
* **큐 시트 (`.cue` 파일)**: 앨범 전체가 하나의 큰 단일 음원 파일(FLAC 등)로 구성되어 있을 때, 트랙별 시간대와 개별 메타데이터를 나열해 둔 텍스트 맵입니다. 플레이어는 내장 태그보다 이 CUE 파일의 정렬 정보를 최우선하여 트랙을 가상으로 분할 파싱합니다.
* **재생 목록 (`.m3u` 파일)**: 재생 리스트 정보가 기록된 파일로, 스캔 시 이 파일이 섞여 있으면 곡 정렬이 파일명이나 트랙 번호 순서가 아닌 `.m3u` 내의 지정된 수열을 따르게 됩니다.
* **로그 파일 (`.log`, `.txt`)**: CD 리핑 신뢰도 검증용 파일로, 메타데이터 파싱 엔진은 이 파일을 감지하더라도 단순히 바이패스(무시)합니다.

## 📦 B2C 음원 배포 표준 (스토어 ➡️ 소비자)
일반 소비자가 다운로드 방식으로 고해상도 음원을 구매하는 환경(Qobuz, Bandcamp, HDtracks 등)에서의 배포 표준 사양은 다음과 같습니다.
* **완전 내장형 태그**: FLAC/ALAC 파일 내부 Vorbis Comment/ID3 영역에 텍스트 정보가 완벽히 메타데이터로 내장됩니다.
* **임베디드 아트워크**: 최소 1400x1400 픽셀 이상의 고해상도 커버 아트 이미지가 음원 파일 헤더에 직접 인베디드(내장) 처리됩니다.
* **폴더 결합형**: 일부 구형 모바일 재생 기기와의 호환성 충돌을 보정하기 위해, 폴더 내 최상단에 별도의 `cover.jpg` 이미지 파일을 동봉하여 제공하는 구조가 표준화되어 있습니다.
